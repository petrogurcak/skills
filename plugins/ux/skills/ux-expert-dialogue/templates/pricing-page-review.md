## SECTION-BY-SECTION REVIEW

### Section 1: Plan Structure

**User provides:**
```
Number of tiers: ____
Tier names: _________________
Free tier: [ ] Yes  [ ] No
Price range: $____ to $____ /month
```

**Expert evaluates:**

**Checklist:**
- [ ] 3-4 tiers maximum (Hick's Law: more choices = slower decisions)
- [ ] Plan names reflect VALUE, not arbitrary labels (not "Bronze/Silver/Gold")
- [ ] Clear progression between tiers (each upgrade is obvious step up)
- [ ] Tier names match target personas (Starter/Pro/Business = clear)
- [ ] No hidden tiers requiring "Contact Sales" without context
- [ ] Feature differences between tiers are meaningful (not just limits)

**Challenge questions:**
1. "Proc mas [N] urovni? Hick's Law rika ze 3-4 jsou optimalni"
2. "Jak se jmenuji tvoje plany a rikaji neco o tom komu jsou urceny?"
3. "Muze uzivatel za 5 sekund pochopit ktery plan je pro nej?"

**Common mistakes:**
- Too many tiers (5+ = decision paralysis, Hick's Law)
- Too few tiers (1-2 = no anchoring, no upsell path)
- Arbitrary names (Gold/Silver = what does that MEAN?)
- Feature differences too subtle between tiers
- Missing a clear "most popular" tier
- Overwhelming feature list per tier

**Data to cite:**
- Hick's Law: Response time increases logarithmically with choices
- 3-tier pricing: +25% conversions vs 5+ tiers (ConversionXL)
- Named personas in tier names: +15% plan identification (Price Intelligently)
- GoodUI: Reducing pricing options from 4 to 3 = +13% purchases

**If too many tiers:**
"Mas [N] planu. Hick's Law: kazda dalsi volba ZPOMALUJE rozhodnuti.

**Optimum:** 3 plany (Good/Better/Best pattern)
- **Starter:** Pro single users, nejnizsi cena, zakladni features
- **Pro:** Pro power users, stredni cena, DOPORUCENY (most popular)
- **Business:** Pro tymy, vyssi cena, all features

**Pravidlo:** Uzivatel musi za 5 sekund identifikovat SVUJ plan."

**Expected impact of fixing:** +13-25% plan selection

```

Number of features listed: ____

```

**Expert evaluates:**

**Checklist:**
- [ ] Comparison table present (not just individual lists)
- [ ] Features grouped by category (not flat list)
- [ ] Most important features at top
- [ ] Tooltips/info icons for complex features
- [ ] Checkmarks vs X marks (not just text)
- [ ] Max 10-15 visible features (expandable for more)
- [ ] Key differentiating features highlighted
- [ ] Feature descriptions in plain language (not technical jargon)

**Challenge questions:**
1. "Je jasne co kazdy feature DELA pro uzivatele? Nebo to jsou technicke termy?"
2. "Jsou features serazeny podle dulezitosti nebo nahodne?"
3. "Kolik features musi uzivatel porovnat? Vice nez 15 = cognitive overload"

**Common mistakes:**
- Flat list of 30+ features (unreadable)
- Technical jargon ("API Rate Limiting: 100/min" - what does that MEAN?)
- No visual differentiation (all text, no checkmarks)
- Random feature order (not by importance)
- No tooltips for complex features
- Same features repeated across all plans (no differentiation visible)

**Data to cite:**
- Miller's Law: 7+-2 items in working memory (group features accordingly)
- Checkmark vs text: +18% scan speed (NNGroup)
- Feature grouping: +23% comparison accuracy (Baymard)
- Tooltips on complex features: +12% understanding (UserTesting)

**If flat unstructured list:**
"Mas [N] features v ploche liste. Miller's Law: uzivatel zvladne 7+-2 polozek.

**Fix:**
1. Seskup features do 3-4 kategorii (Core / Advanced / Enterprise)
2. Top 5 features nahore (to co rozhoduje)
3. Checkmarks misto textu pro rychle skenovani
4. Tooltip ikonky u komplexnich features
5. 'Zobrazit vice' pro 15+ features

**Result:** +23% presnost porovnani, +18% rychlost skenovani."

**Expected impact of fixing:** +18-23% comparison effectiveness

### Section 4: Price Display

**User provides:**
```
Price format: $____ /month
Annual option: [ ] Yes, savings shown  [ ] Yes, no savings  [ ] No
Toggle: [ ] Monthly/Annual toggle  [ ] Separate pages  [ ] No toggle
Anchoring: [ ] Crossed-out price  [ ] "Save X%"  [ ] None
Per-unit pricing: [ ] Per user  [ ] Per seat  [ ] Flat  [ ] Custom
```

**Expert evaluates:**

**Checklist:**
- [ ] Monthly/Annual toggle present
- [ ] Annual savings clearly shown (% or $ amount)
- [ ] Annual option pre-selected (default = annual for higher LTV)
- [ ] Price anchoring used (crossed-out higher price)
- [ ] Per-unit pricing clear for seat-based models
- [ ] Currency appropriate for visitor's location
- [ ] No cent amounts for plans >$10 ($49 not $49.00)
- [ ] Price size hierarchy (price is largest text in plan card)

**Challenge questions:**
1. "Ukazujes usporu pri rocni platbe? Je to procento nebo castka?"
2. "Ktery billing je defaultne vybrany? Annual by mel byt default"
3. "Pouzivas price anchoring? Preskrtla cena zvysuje vnimany deal o 20-30%"

**Common mistakes:**
- No annual option (losing LTV)
- Annual savings not quantified ("Save more!" vs "Save $120/year")
- Monthly pre-selected (lower LTV)
- No price anchoring (user doesn't feel they're getting a deal)
- Confusing per-seat math (user must calculate team cost themselves)
- Cent amounts ($49.99 looks cheap, $49 looks premium)

**Data to cite:**
- Annual pre-selected: +20% annual plan adoption (Price Intelligently)
- Showing exact savings: +15% annual conversion ("Save $120" vs "Save 20%")
- Price anchoring (crossed-out): +20-30% perceived value (Dima Melnik)
- Charm pricing ($9 vs $10): +24% for low-price items, irrelevant for SaaS (MIT study)
- Removing cents from SaaS pricing: +8% premium perception

**If no annual toggle:**
"Chybi rocni moznost. To stoji LTV.

**Fix:**
1. Pridej monthly/annual toggle
2. Default = annual (vyssi LTV)
3. Ukaz presnou usporu: 'Usetrte $120/rok' (ne jen '20% sleva')
4. Preskrtni mesicni cenu u annual ($49 ~~$59~~/mesic)

**Math:**
- Monthly: $59 x 12 = $708/year per user
- Annual: $49 x 12 = $588/year per user
- Savings per user: $120/year
- Pokud 20% switche na annual = +20% LTV immediately"

**Expected impact of fixing:** +15-30% annual plan adoption

### Section 5: CTA Per Plan

**User provides:**
```
CTA text per plan: _________________
CTA style: [ ] Same for all  [ ] Different per plan  [ ] Highlighted for recommended
Trial offered: [ ] Yes  [ ] No
CTA text examples: _________________
```

**Expert evaluates:**

**Checklist:**
- [ ] Each plan has distinct CTA
- [ ] Recommended plan CTA is most prominent (filled button)
- [ ] Other plans have secondary CTA style (outlined button)
- [ ] CTA text reflects plan value (not just "Sign Up")
- [ ] Free trial CTA is risk-free ("Start Free Trial" not "Buy Now")
- [ ] CTA includes benefit hint ("Start Growing" not "Select Plan")
- [ ] No credit card required for trial (if applicable, state it)
- [ ] Enterprise CTA is "Talk to Sales" (not "Buy")

**Challenge questions:**
1. "Jsou CTA texty specificke pro kazdy plan nebo vsude 'Sign Up'?"
2. "Vyzaruji CTA duveru? 'Start Free Trial' vs 'Buy Now' je velky rozdil"
3. "Je jasne ze trial nevyzaduje kreditku? To zvysuje konverzi o 30%"

**Common mistakes:**
- Same "Sign Up" CTA for all plans (no differentiation)
- "Buy Now" for trial plans (too aggressive)
- All CTAs same visual weight (no guidance)
- Missing "No credit card required" (fear barrier)
- Enterprise "Buy" instead of "Talk to Sales"
- Generic text that doesn't hint at value

**Data to cite:**
- Specific CTA text: +213% vs generic (Michael Aagaard, ContentVerve)
- "No credit card required": +30% trial signups (Totango SaaS benchmarks)
- Primary vs secondary CTA styling: +22% click on recommended plan (VWO)
- First-person CTA ("Start MY trial"): +90% vs second-person (Unbounce)

**If generic CTAs:**
"Vsechny plany maji 'Sign Up'. Zero differentiation.

**Fix:**
- Starter: 'Start Free' (low commitment, outlined button)
- Pro: 'Start My Free Trial' (first-person, FILLED button, largest)
- Business: 'Start Team Trial' (team-specific)
- Enterprise: 'Talk to Sales' (different action)

**Plus:** Pod CTA button: 'Bez kreditni karty. Zruseni kdykoliv.'

**Expected:** +90% z first-person CTA + 30% z 'no credit card' = significant lift."

**Expected impact of fixing:** +22-90% CTA click-through
